# مؤسسة شادي الملكاوي — نظام إدارة المبيعات والمستودع

نظام ويب عربي لإدارة الطلبات، العملاء، المخزون، المشتريات، المرتجعات، التقارير، المستخدمين، وسجل العمليات. يعمل بواجهة واحدة مبنية بـ HTML/CSS/Vanilla JavaScript مع Firebase Realtime Database، بدون React وبدون Vite وبدون TypeScript.

---

## 1. نظرة عامة

النظام مصمم لإدارة دورة الطلب كاملة من لحظة إدخال الطلب حتى التسليم، مع تتبع المخزون والباركود، إدارة العملاء، تقارير تشغيلية، وصلاحيات مستخدمين.

أهم محاور النظام:

- تسجيل طلب جديد بخطوات واضحة.
- إدارة الطلبات حسب الحالة.
- إدارة المخزون والكميات والباركود.
- تسجيل المشتريات والمرتجعات.
- تقارير وبحث وتصدير Excel.
- CRM للعملاء.
- إعدادات الموقع: التعريفات، الألوان، المستخدمون، والصلاحيات.
- سجل موحّد يجمع سجل العمليات وسجل التدقيق المالي.
- دعم العمل دون اتصال جزئياً من خلال Offline Queue و Service Worker.
- واجهة عربية RTL متجاوبة مع الجوال والتابلت واللابتوب.

## 2. بنية الملفات الحالية

| الملف | الوصف |
|---|---|
| `index.html` | هيكل الواجهة الكامل، الصفحات، النوافذ، شاشات الدخول، والتنقل. |
| `app.js` | منطق التطبيق بالكامل: الدخول، الطلبات، المخزون، التقارير، المستخدمون، السجلات، Firebase. |
| `constants.js` | الثوابت والقوائم الأساسية مثل الحالات والألوان والتصنيفات. |
| `firebase-config.js` | إعدادات Firebase ومراجع قواعد البيانات. |
| `style.css` | ملف التصميم الوحيد المعتمد حالياً، ويحتوي كل قواعد التصميم بعد دمج `style-shm.css` داخله. |
| `sw.js` | Service Worker للكاش والتحديث والعمل الجزئي دون اتصال. |
| `version.json` | رقم إصدار النشر الحالي. |
| `SHM.png` | شعار النظام و favicon. |
| `barcode-replace.html` | أداة/صفحة مساعدة مرتبطة بالباركود. |
| `README.md` | هذا الملف. |

> تم إلغاء الاعتماد على `style-shm.css`. التصميم الآن يعمل من ملف واحد فقط: `style.css`.

## 3. الصفحات الأساسية داخل النظام

### 3.1 لوحة القيادة
تعرض ملخصاً تنفيذياً سريعاً عن الطلبات، الإيرادات، المخزون، والتنبيهات.

### 3.2 الطلبات
تعرض الطلبات على شكل بورد حسب الحالة مع إجراءات المتابعة، العرض، الحذف، واتساب، وتغيير الحالة.

### 3.3 طلب جديد
واجهة إدخال الطلبات الجديدة بخطوات، وتدعم اختيار العميل، المحافظة، المنتج، اللون، المقاس، الكمية، السعر، المصدر، والمدخل.

### 3.4 التقارير
صفحة بحث وتصفية للطلبات حسب الحالة، المنتج، المحافظة، التاريخ، والنص الحر، مع إمكانية التصدير إلى Excel.

### 3.5 المستودع
إدارة المنتجات والكميات والباركود وتتبع المخزون. تدعم عرض حركة الصنف والتعديلات المخزنية.

### 3.6 المشتريات
تسجيل دخول بضاعة جديدة للمستودع مع دعم الباركود والأسعار والصفحة/المصدر.

### 3.7 المرتجعات
تسجيل المرتجعات وربطها بالمخزون لتحديث الكميات تلقائياً.

### 3.8 العملاء
CRM مدمج يعرض العملاء، أرقامهم، المحافظات، الإيرادات، عدد الطلبات، آخر طلب، وتصنيف العميل.

### 3.9 إعدادات الموقع
صفحة موحدة تجمع:

- الصفحات / المصادر.
- الألوان المخصصة.
- المستخدمين والصلاحيات.
- إدارة أدوار المستخدمين.
- إنشاء مستخدم جديد.
- فلترة المستخدمين حسب الاسم أو الدور.

تم دمج صفحات `التعريفات` و`المستخدمون` في هذه الصفحة.

### 3.10 السجل والتدقيق
صفحة موحدة تجمع:

- سجل العمليات العام.
- سجل التدقيق المالي.
- حركات إنشاء/تعديل/حذف الطلبات.
- تغييرات الأسعار والمخزون.
- المستخدم المنفذ.
- المرجع.
- التفاصيل والفروقات قبل/بعد.

تم دمج صفحات `السجل` و`سجل التدقيق` في صفحة واحدة، وتم وضعها في نهاية شريط التنقل.

## 4. تعديلات التنظيف الأخيرة

تم تنفيذ تنظيف حذر مع الحفاظ على شكل الموقع:

1. دمج `السجل` و`سجل التدقيق` في صفحة واحدة باسم `السجل والتدقيق`.
2. دمج حركات السجلين داخل جدول موحد مع فلاتر للمصدر، الحركة، المستخدم، والبحث.
3. إبقاء صفحة `السجل والتدقيق` في آخر شريط التنقل.
4. حذف زر `مستودع المؤسسة` من شريط التنقل بالكامل.
5. دمج `التعريفات` و`المستخدمون` في صفحة واحدة باسم `إعدادات الموقع`.
6. دمج `style.css` و`style-shm.css` في ملف تصميم واحد هو `style.css`.
7. حذف رابط `style-shm.css` من `index.html`.
8. حذف `style-shm.css` من ملفات المشروع.
9. تحديث `sw.js` لإزالة `style-shm.css` من الكاش.
10. تحديث رقم الإصدار في `version.json`.

## 5. نظام الدخول وتذكّر الحساب

النظام يحتوي شاشة دخول مخصصة:

- اسم مستخدم.
- كلمة مرور.
- خيار `تذكّرني`.
- عرض الحسابات المحفوظة.
- دخول سريع للحساب المحفوظ.

ملاحظة أمنية: حفظ كلمة المرور محلياً يتم بصيغة مموّهة محلياً، وليس بديلاً عن نظام مصادقة مؤسسي متقدم. يجب استخدامه فقط على أجهزة موثوقة.

أهم المسارات المستخدمة منطقياً:

- الطلبات.
- المخزون.
- المرتجعات.
- المشتريات.
- التعريفات.
- المستخدمون.
- السجلات.
- التدقيق المالي.
- الألوان المخصصة.

يجب عدم تغيير أسماء المسارات دون مراجعة `app.js` و`firebase-config.js` معاً.

---

## 8. Service Worker والكاش

الملف `sw.js` يستخدم سياسة Network-First لملفات الموقع، ويحدث الكاش تلقائياً عند رفع نسخة جديدة.

الإصدار الحالي للكاش:

```js
shm-cache-v7
```

عند تعديل ملفات رئيسية مثل:

- `index.html`
- `app.js`
- `style.css`
- `constants.js`
- `firebase-config.js`

يفضل رفع رقم الكاش داخل `sw.js` لضمان وصول التحديث للمستخدمين.

## 12. قواعد مهمة قبل أي تعديل جديد

قبل أي تعديل جديد:

1. لا تغيّر أسماء `id` المستخدمة في `app.js` إلا إذا عدلت كل المراجع.
2. لا تحذف عناصر HTML قبل البحث عن استخدامها داخل `app.js`.
3. لا تغير مسارات Firebase بدون تعديل `firebase-config.js` والمنطق المرتبط.
4. لا تضف ملف CSS ثاني؛ التصميم أصبح موحداً داخل `style.css`.
5. بعد أي تعديل في الملفات الأساسية، حدّث `version.json` ويفضل تحديث `CACHE_NAME` داخل `sw.js`.
6. افحص `app.js` بـ `node --check app.js` قبل التسليم.
7. لا تضف React أو TypeScript أو Tailwind أو أي مكتبة UI.

نفّذ الفحص التالي محلياً:


ثم افتح `index.html` وتحقق من:

- تسجيل الدخول.
- تذكّرني.
- التنقل بين الصفحات.
- إدخال طلب جديد.
- فتح المستودع.
- فتح إعدادات الموقع.
- فتح السجل والتدقيق.
- ظهور الجداول بتباين واضح.
- عمل زر FAB.
- عدم ظهور زر مستودع المؤسسة.

## 14. ملاحظات تشغيلية

- المشروع لا يحتاج Build Step.
- يمكن تشغيله كملفات Static على GitHub Pages أو أي استضافة ثابتة.
- يحتاج اتصال Firebase صحيح حتى تعمل البيانات الحية.
- عند استخدام GitHub Pages، تأكد أن الدومين مضاف ضمن Authorized Domains في Firebase عند الحاجة.
